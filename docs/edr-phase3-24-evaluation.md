# MxSec EDR Phase 3-24 商业级评估方案

**评估人**: Claude (Anthropic)
**日期**: 2026-05-21
**前提**: Phase 1（自研 eBPF 采集层）+ Phase 2（Agent 端规则引擎）已完成并合并 dev

---

## 一、整体评判：24 Phase 路线图的结构性问题

### 1.1 核心问题：Phase 粒度不一致，依赖链过长

设计文档将 24 个 Phase 按"功能模块"线性排列，但商业产品开发不是线性的。关键问题：

| 问题 | 说明 |
|------|------|
| **粒度差异巨大** | Phase 5 (YARA 联动) 1-2 周 vs Phase 21 (SOC 前端) 4-6 周 vs Phase 23 (Windows) 6-8 周 |
| **依赖链未显式声明** | Phase 11 (故事线) 依赖 Phase 4 (进程树) + Phase 10 (BDE)，但排列上看不出来 |
| **价值交付间隔过长** | 按线性执行，Phase 8 (K8s) 要等约 20 周，但可能是客户最急需的 |
| **没有 MVP 切片** | 每个 Phase 都是"完整实现"，没有区分"最小可交付"vs"完整能力" |

### 1.2 建议：重组为 4 个交付里程碑 (Milestone)

放弃 Phase 3→4→5→...→24 线性模式，改为**按商业价值交付的里程碑**，每个里程碑包含多个 Phase 的核心子集，交付后即可对外演示/试用：

```
M1: 检测闭环 (Agent→Server→告警→响应)     ← 最小可演示 EDR
M2: 深度检测 (进程树+IOC+YARA+序列)         ← 可比肩开源竞品
M3: 企业级运维 (K8s+灰度+隔离+WAL)         ← 可试点部署
M4: 差异化能力 (BDE+故事线+MQL+ML)          ← 商业竞争力
```

## 二、逐 Phase 评估

### Phase 3: 规则生态 (设计 3-4 周)

**商业价值**: ★★★★☆ — 社区仓库+规则同步是开源 EDR 差异化的核心
**技术风险**: ★★☆☆☆ — 主要是工程量，无技术难点
**依赖**: Phase 2（已完成）

**评估**:
- 3.1 社区仓库初始化：**M1 需要，但最小化**。v1 不需要独立 Git 仓库，`configs/rules/` 目录内嵌即可
- 3.2 CEL→YAML 迁移：**推迟**。现有 38 条 CEL 规则仍然有效，YAML 是新增能力不是替代
- 3.3 Server 规则同步（Git pull）：**M2**。M1 只需内嵌规则
- 3.4 Agent 规则订阅（Heartbeat 增量下发）：**M1 必须**。没有下发机制，规则引擎是死的
- 3.5 规则测试 CLI：**M2**。非阻塞，社区贡献前才需要
- 3.6-3.7 测试数据集+CI：**M2**

**裁剪建议**:
- M1 只做 3.4（规则下发）+ 内嵌 20-30 条核心 YAML 规则
- M2 做 3.1+3.3+3.5+3.6

### Phase 4: Server 端引擎增强 (设计 3-4 周)

**商业价值**: ★★★★★ — 没有进程树和 IOC 接入，检测深度停留在 Level 1
**技术风险**: ★★★☆☆ — 进程树内存管理需要仔细设计
**依赖**: Phase 1（/proc 快照已完成）

**评估**:
- 4.1 IOC 接入检测流水线：**M1 必须**。abuse.ch 数据已在 Redis，只差接入
- 4.2 进程树构建：**M1 必须**。Agent 已有 /proc 快照上报，Server 端只需消费构建
- 4.3 CEL 自定义函数：**M1 部分**。`ioc_match` + `ancestor_has` 是刚需，`count_recent`/`first_seen` 延 M2
- 4.4 告警聚合去重：**M1 必须**。没有去重 = 告警风暴 = 产品不可用
- 4.5 序列检测持久化：**M2**
- 4.6 CEL 并行优化：**M3**。1000 主机前不是瓶颈
- 4.7 行为基线学习：**M4**。依赖 BDE

**裁剪建议**:
- M1 做 4.1+4.2+4.3(部分)+4.4
- M2 做 4.3(完整)+4.5
- M3 做 4.6
- M4 做 4.7

### Phase 5: YARA 联动 (设计 1-2 周)

**商业价值**: ★★★★☆ — 文件恶意检测是 EDR 标配
**技术风险**: ★★★☆☆ — yara-x Go binding 成熟度需验证；pre-block (SIGSTOP) 对生产有风险
**依赖**: Phase 2（规则 YAML 的 yara 字段）

**评估**:
- 5.1 内嵌 yara-x：**M2**。编译依赖 + 内存开销需要评估
- 5.2 async 模式：**M2 必须**。先放行后扫描是安全默认
- 5.3 pre-block 模式：**M3**。SIGSTOP/SIGCONT 在生产中有风险（父进程可能检测到子进程停止），需要充分灰度
- 5.4-5.5：**M2**

**风险提醒**:
- yara-x 是 Rust 实现，Go binding 通过 CGO，会增加编译复杂度和二进制大小（+10-20MB）
- 建议评估 go-yara (纯 CGO binding to libyara) 作为备选
- pre-block 模式建议增加白名单机制：对 systemd、sshd、agent 自身等关键进程永远不 SIGSTOP

### Phase 6: 事件采集增强 (设计 2-3 周)

**商业价值**: ★★★☆☆ — DNS 事件对 C2 检测价值极高，但 rename/chmod 价值一般
**技术风险**: ★★★★☆ — BPF 内 DNS payload 解析是新的复杂度域
**依赖**: Phase 1（已完成 eBPF 框架）

**评估**:
- 6.1 DNS 事件：**M2**。C2 通信和 DNS 隧道检测的基础。但 BPF 内解析 DNS 包需要处理 TCP/UDP 分片、压缩指针等，复杂度高。建议 M2 先做简单版（仅 UDP port 53，不解析响应），M3 做完整版
- 6.2 file_rename/unlink/chmod：**M2**。勒索软件检测需要 rename 事件（大量重命名=加密），chmod 对持久化检测有用
- 6.3 cwd/tty/env 字段：**M1**。进程上下文完整度直接影响规则质量。cwd 从 BPF 读取较简单，env 需要限制长度
- 6.4 网络 bytes/direction：**M3**。数据渗出检测需要，但非核心
- 6.5 eBPF 信号监控：**M3**。自保增强
- 6.6 用户态高级规则禁用：**已完成**（Phase 1.5 capability 体系）

**风险提醒**:
- DNS BPF 解析的 verifier 限制是最大风险。DNS 包格式不固定长度，BPF 验证器对循环和变长访问限制严格。建议备选方案：用户态在 perf buffer 中拿到原始 UDP payload 后解析

### Phase 7: 事件可靠性与数据安全 (设计 2-3 周)

**商业价值**: ★★★★☆ — WAL 是生产部署的硬要求，丢事件=检测盲区
**技术风险**: ★★★☆☆ — WAL 实现有成熟模式，但崩溃恢复测试是关键
**依赖**: 无（独立模块）

**评估**:
- 7.1 WAL 缓冲：**M3 必须**。网络断开不丢事件是企业级底线
- 7.2 事件排序：**M3**。host_id Kafka Key 已隐式保证单主机有序
- 7.3 mTLS 轮换：**推迟**。当前已有 mTLS（90 天证书），自动轮换是运维优化
- 7.4 规则签名验证：**M3**。防止中间人篡改规则
- 7.5 敏感字段脱敏：**M2**。cmdline 中的密码/token 泄露是合规风险

**风险提醒**:
- WAL 与当前 RingBuffer 的关系需要明确。当前 transport 用 RingBuffer 缓冲，WAL 是替代还是补充？建议：RingBuffer 作为热缓冲（内存），WAL 作为持久化补充（磁盘），溢出时写 WAL
- WAL 文件格式建议用 Protocol Buffers 而非自定义格式，便于调试和工具支持

### Phase 8: 容器与 K8s 环境 (设计 3-4 周)

**商业价值**: ★★★★★ — K8s 是客户最关心的场景，不支持容器=丢单
**技术风险**: ★★★☆☆ — 多容器运行时适配是工程量
**依赖**: Phase 1（container_id 提取已在 eBPF 中预留）

**评估**:
- 8.1 ContainerRuntime 适配器：**M3 必须**。containerd 优先（K8s 默认）
- 8.2 容器 ID 提取：**部分已完成**。Phase 1 eBPF 已有 `in_container` 标记和 cgroup 解析
- 8.3 K8s 上下文注入：**M3**。pod/namespace/labels 注入
- 8.4 容器逃逸检测规则：**M3**。5-10 条核心规则
- 8.5 DaemonSet + Helm：**M3**

**风险提醒**:
- 容器内 Agent 和宿主机 Agent 是同一个二进制。DaemonSet 模式下 Agent 以特权容器运行在宿主机上，不是容器内运行。这点设计文档已正确描述
- Kubelet API 10255（只读端口）在新版 K8s 默认关闭，需要用 10250 + ServiceAccount Token 认证

### Phase 9: 运维体系 (设计 2-3 周)

**商业价值**: ★★★★☆ — 灰度升级+熔断器是生产必备
**技术风险**: ★★☆☆☆ — 成熟模式
**依赖**: Phase 3（规则下发）

**评估**:
- 9.1 Agent 生命周期：**部分已完成**。注册/升级/自更新在 Phase 1 前已有实现
- 9.2 灰度升级：**M3**。分批 + 自动回滚
- 9.3 健康心跳体系：**M1 部分已完成**。Phase 1.8 已有 EDR 能力上报
- 9.4 管理后台大盘：**M3**。前端工作
- 9.5 响应动作熔断器：**M1 必须**。没有熔断器=一条规则误报可以杀掉整个集群的进程
- 9.6 规则灰度发布：**M3**

**裁剪建议**:
- M1 只做 9.5（熔断器）
- M3 做 9.2+9.4+9.6

### Phase 10: 行为检测引擎 BDE (设计 3-4 周)

**商业价值**: ★★★★★ — 商业 EDR 核心差异化，检测未知威胁
**技术风险**: ★★★★★ — 行为基线的统计模型需要大量调参，误报控制是核心难题
**依赖**: Phase 4（进程树）+ Phase 6（完整事件字段）

**评估**:
- 10.1 Agent 端行为采集：**M4**。4 维统计 + 滑动窗口
- 10.2 Server 端基线引擎：**M4**。偏差计算 + risk_score
- 10.3 跨主机全局基线：**M4**
- 10.4 BDE 告警集成：**M4**
- 10.5 用户态降级（3 维）：**M4**

**风险提醒**:
- BDE 的核心挑战不是实现，是调参。行为基线的 σ（标准差）对不同类型进程差异巨大。nginx worker 的行为非常稳定（σ 小），但 cron job 的行为天然波动大（σ 大）
- 建议先做 "离线分析模式"：收集 30 天真实数据 → 离线跑基线算法 → 调参 → 再上线实时引擎
- 冷启动问题：新 Agent 7 天内无基线，建议用"全局基线"（同类主机的聚合基线）作为初始值

### Phase 11: 攻击故事线引擎 (设计 3-4 周)

**商业价值**: ★★★★★ — SentinelOne 核心卖点，也是 SOC 分析师最需要的
**技术风险**: ★★★★☆ — story_id 传播 + 跨主机关联是分布式系统问题
**依赖**: Phase 4（进程树）+ Phase 10（BDE，可选）

**评估**:
- 11.1 Agent 端 CausalTracker：**M4**
- 11.2 Server 端故事线聚合：**M4**
- 11.3 跨主机关联：**M4+**。SSH 横向移动场景的跨主机关联非常复杂
- 11.4 故事线可视化：**M4**（前端）
- 11.5 故事线 API：**M4**

**风险提醒**:
- story_id 在进程树中传播看似简单，但有边界情况：
  - fork+exec 后 story_id 是否继承？（应该继承）
  - cron 任务的 story_id 是每次新建还是复用？（应该每次新建）
  - SSH 远程命令的 story_id 如何跨主机传递？（需要 Agent→Server→Agent 三跳）
- 建议 M4 先做单主机故事线，跨主机关联推迟到 M4+

### Phase 12: 威胁狩猎 MQL (设计 3-4 周)

**商业价值**: ★★★★☆ — SOC 分析师日常工具，但需要 ClickHouse 有足够数据
**技术风险**: ★★★★☆ — SQL 编译器 + 安全护栏是较复杂的工程
**依赖**: Phase 7（事件存储完善）

**评估**: **M4**。整体推迟到 M4。短期内直接用 ClickHouse SQL 查询 + Grafana 看板可以替代。

**替代方案建议**:
- M3 做一个简单的"事件搜索 API"（时间范围+字段过滤+分页），前端搜索框即可
- MQL 编译器是大投入，且需要用户教育成本。先做搜索，后做 MQL

### Phase 13: 自动编排 Playbook (设计 3-4 周)

**商业价值**: ★★★☆☆ — 企业级需求，但需要告警体系成熟后才有意义
**技术风险**: ★★★☆☆ — 状态机 + 持久化 + 审批流是成熟模式
**依赖**: Phase 4（告警体系）+ Phase 9（熔断器）

**评估**: **M4+**。Playbook 需要告警体系、响应动作、熔断器全部就绪才有意义。过早实现=空壳。

### Phase 14: 主机网络隔离 (设计 2-3 周)

**商业价值**: ★★★★☆ — 应急响应核心能力
**技术风险**: ★★★★★ — 网络隔离做错了=主机失联=P0 事故
**依赖**: Phase 1（eBPF 框架）

**评估**: **M3**。但必须极其保守：
- 默认只有 alert + 建议隔离，不自动执行
- enforce 模式需要管理员双重确认（API + 短信/邮件验证码）
- 必须保留管控通道（Agent↔Server），否则隔离后无法解除

**风险提醒（严重）**:
- 设计文档提到 `block_ip` action，Phase 2 评估时已推迟。网络隔离比 block_ip 更危险
- eBPF cgroup/connect 隔离如果 attach 到错误的 cgroup → 影响所有容器
- 建议 M3 只实现 iptables/nftables 方案（可逆、可审计），eBPF 隔离推迟到 M4

### Phase 15: 内存威胁检测 (设计 2-3 周)

**商业价值**: ★★★☆☆ — 无文件攻击检测是高级能力，但频率不高
**技术风险**: ★★★★☆ — memfd 监控简单，但 /proc/pid/mem YARA 扫描有性能风险
**依赖**: Phase 5（YARA）+ Phase 6（eBPF 增强）

**评估**: **M4**。无文件攻击在实际生产中占比 < 5%，优先级低于其他 Phase。

### Phase 16: 大规模部署与高可用 (设计 2-3 周)

**商业价值**: ★★★★☆ — 企业客户试点后扩量的前提
**技术风险**: ★★☆☆☆ — 主要是 Kafka/Redis/ClickHouse 运维配置
**依赖**: 无（基础设施层面）

**评估**: **M3**。水平扩展+离线支持+多租户预留。

### Phase 17: ML 检测引擎 (设计 4-6 周)

**商业价值**: ★★★★☆ — 静态 ML 检测恶意 ELF 是差异化能力
**技术风险**: ★★★★★ — 模型训练需要标注数据集，ONNX Runtime 增加 Agent 内存
**依赖**: 无（独立模块）

**评估**: **M4+**。ML 模型需要：
1. 足够的训练数据（EMBER 数据集 + 自采集恶意样本）
2. 持续的模型迭代（月度重训）
3. 对抗样本防护

**风险提醒**:
- ONNX Runtime Go binding 增加约 30-50MB 内存，Agent 总 RSS 目标 50MB 可能不够
- 建议先做 Server 端 ML（不受 Agent 内存限制），Agent 端只做特征提取上报

### Phase 18: 文件信誉与沙箱 (设计 3-4 周)

**商业价值**: ★★★☆☆ — 需要外部沙箱基础设施
**技术风险**: ★★★☆☆ — Bloom Filter + LRU 成熟模式
**依赖**: Phase 5（YARA 联动）

**评估**: **M4+**。沙箱对接需要 CAPE/Cuckoo 部署，非自研范围。先做文件信誉本地缓存即可。

### Phase 19: 多源威胁情报 (设计 2-3 周)

**商业价值**: ★★★★☆ — 多源 IOC 交叉验证提升置信度
**技术风险**: ★★☆☆☆ — 适配器模式
**依赖**: Phase 4（IOC 接入流水线）

**评估**: **M3**。当前只有 abuse.ch 单源，扩展到 OTX/MISP 等。

### Phase 20: 远程取证与响应 (设计 2-3 周)

**商业价值**: ★★★★☆ — 应急响应核心能力
**技术风险**: ★★★★☆ — 远程 Shell 安全性是最大挑战
**依赖**: Phase 7（mTLS 加固）

**评估**: **M3**。但安全要求极高：
- 远程 Shell 必须有完整审计录制
- 必须有危险命令拦截（rm -rf /、dd if=/dev/zero 等）
- 必须有会话超时和强制断开

### Phase 21: SOC 级前端 (设计 4-6 周)

**商业价值**: ★★★★★ — 产品的"脸面"，决定第一印象
**技术风险**: ★★☆☆☆ — 前端工程，无技术难点
**依赖**: 后端 API 就绪

**评估**: 分散到各里程碑：
- M1: 告警列表+基础仪表盘
- M2: 规则管理+事件搜索
- M3: 主机详情+K8s 视图+隔离管理
- M4: 故事线可视化+威胁狩猎+BDE 大盘

### Phase 22: 开放 API + 生态集成 (设计 2-3 周)

**商业价值**: ★★★☆☆ — 企业级需求，SIEM 对接
**技术风险**: ★☆☆☆☆
**依赖**: 核心功能就绪

**评估**: **M3**。RESTful API + SIEM Syslog CEF + RBAC 增强。

### Phase 23: Windows 支持 (设计 6-8 周)

**商业价值**: ★★★★★ — 企业环境 Windows 占比 > 50%
**技术风险**: ★★★★★ — 全新技术栈（ETW、Minifilter、AMSI、PE 分析）
**依赖**: 核心引擎架构稳定

**评估**: **M4+**。工作量最大的单个 Phase。建议：
- M4 先做 Windows Agent 骨架（ETW 事件采集 + 基础规则匹配）
- M4+ 做 PE ML + Minifilter + AMSI

**风险提醒**:
- ETW 和 eBPF 是完全不同的技术范式，不能简单移植
- Minifilter 驱动需要 Windows Driver 开发经验，可能需要外部协助
- 建议先用纯 ETW 方案（无驱动），覆盖 80% 场景

### Phase 24: 规则扩充 (持续迭代)

**评估**: 贯穿所有里程碑，不单独排期。每个里程碑交付时补充对应的检测规则。

## 四、关键架构决策建议

### 4.1 规则下发机制（M1 最关键决策）

设计文档描述了 Heartbeat 增量下发，但细节不足。建议：

```
方案 A: Heartbeat 轮询式（简单）
  Agent 心跳上报 rules_version → Server 对比 → 不一致时通过 gRPC 双向流推送规则包
  优点: 复用现有 Heartbeat 通道
  缺点: 最大延迟 = 心跳间隔（30s）

方案 B: gRPC Server Push（实时）
  Server 维护 Agent 连接表，规则变更时主动推送
  优点: 实时
  缺点: Server 需维护长连接状态

推荐: 方案 A。30s 延迟完全可接受，且复用现有架构。
```

### 4.2 熔断器设计（M1 安全底线）

```
三级熔断:
1. 规则级: 单规则 5min 内 kill > 50 → 该规则降级为 alert
2. 主机级: 单主机 5min 内 kill > 20 不同进程 → 该主机暂停所有 kill
3. 全局级: 全局 5min 内 kill > 200 → 所有 Agent 暂停 kill

实现位置:
- 规则级: Agent 端（本地计数器）
- 主机级: Agent 端
- 全局级: Server 端（通过 Heartbeat 下发熔断指令）

恢复:
- 自动: 30min 后自动恢复
- 手动: 管理员 API 确认后恢复
```

### 4.3 进程树内存管理（M1 必须解决）

```
设计文档估算: 200B/node × 500进程/主机 × 1000主机 ≈ 100MB
实际可能更高: 字符串字段（exe, cmdline, ancestor_chain）平均 200-500B

建议:
- ProcessNode 使用 string intern（相同 exe 路径共享内存）
- cmdline 截断到 4KB
- TTL: 进程退出后保留 2h（用于延迟到达的事件关联），然后清理
- 内存硬限制: 进程树总内存 > 500MB 时强制驱逐最旧节点
```

### 4.4 M1 需要的前端改动

```
最小化前端改动（M1）:
1. 告警列表页（新增页面）
   - 表格: 时间/主机/规则/严重级/状态/操作
   - 筛选: 严重级/时间范围/主机
   - 操作: 查看详情/标记处理/白名单

2. 仪表盘（现有页面增强）
   - 新增卡片: 今日告警数/Critical 数/Agent 在线率
   - 新增趋势图: 24h 告警趋势

3. 主机详情页（增强）
   - 新增 Tab: EDR 事件流（最近 100 条）
   - 新增字段: 采集模式/规则数/降级级别
```

## 五、与设计文档的分歧总结

| 设计文档 | 本评估建议 | 原因 |
|----------|-----------|------|
| 24 Phase 线性执行 | 4 Milestone 交叉执行 | 价值交付更快，依赖关系更清晰 |
| Phase 3 社区仓库优先 | M1 先内嵌规则，M2 做社区仓库 | 社区仓库需要治理成本，内嵌规则零成本 |
| Phase 5 pre-block 与 async 同时实现 | M2 先 async，M3 再 pre-block | pre-block 有生产风险，需要灰度验证 |
| Phase 6 DNS BPF 内核解析 | M2 先用户态解析，M3 做内核解析 | BPF verifier 对 DNS 解析限制多 |
| Phase 7 WAL 替换当前缓冲 | WAL 作为 RingBuffer 的持久化补充 | 保留内存热缓冲的低延迟优势 |
| Phase 12 MQL 编译器 | M3 先做事件搜索 API，M4 做 MQL | MQL 投入产出比低，搜索 API 覆盖 80% |
| Phase 14 eBPF 隔离优先 | M3 先 iptables，M4 做 eBPF | iptables 可逆可审计，eBPF 隔离风险高 |
| Phase 17 Agent 端 ML | Server 端 ML 优先，Agent 端 ML 延后 | Agent 内存预算不足 |
| block_ip 作为规则 action | 推迟到 M4（eBPF XDP/TC） | Phase 2 评估已确认：iptables 篡改风险 |
| Watchdog 双进程互保 | systemd WatchdogSec 足够（已实现） | 双进程增加攻击面和资源消耗 |

## 六、实施风险矩阵

| 风险 | 影响 | 概率 | 缓解 |
|------|------|------|------|
| 进程树内存爆炸（>1GB） | Server OOM | 中 | 内存硬限制+驱逐策略+string intern |
| 熔断器缺失导致批量误杀 | 客户业务中断 | 高（无熔断时） | M1 必须实现三级熔断 |
| yara-x CGO 编译问题 | Agent 无法构建 | 中 | 备选 go-yara + 独立构建验证 |
| DNS BPF 解析 verifier 拒绝 | DNS 事件不可用 | 中 | 用户态解析 fallback |
| K8s Kubelet API 认证失败 | 容器上下文缺失 | 低 | 三级获取策略（Downward API → Kubelet → /proc） |
| WAL 文件损坏导致数据丢失 | 检测盲区 | 低 | CRC 校验 + 段级别恢复 |
| 网络隔离误操作导致主机失联 | P0 事故 | 中 | 管控通道白名单 + 双重确认 + 超时自动解除 |
| BDE 冷启动期误报 | 告警风暴 | 高 | 冷启动 risk_score ×0.5 + critical 规则不受影响 |
